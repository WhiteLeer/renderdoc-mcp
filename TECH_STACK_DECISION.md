# renderdoc-workbench GUI 技术栈建议

## 1. 目标

在真正开工前，先把第一期 GUI 技术栈建议收口，避免边写边摇摆。

当前不是最终拍板文档，但会给出明确推荐。

## 2. 当前仓库现状

从当前仓库结构可以直接确认：

- 根目录已有 Qt5 DLL
- 仓库中已有 `PySide2/`
- 当前运行时显然已经偏向 Python + Qt 体系

同时也要注意两个实际问题：

- 当前仓库根目录自带的 `PySide2` 与 `python36.dll` 绑定
- 当前仓库内置的 `python313` 运行时已经可以安装并使用 `PySide6`

这说明两件事要分开看：

- 启动策略必须优先绑定仓库自己的 Python 运行时
- GUI 具体使用哪套 Qt 绑定，还需要单独收口

## 3. 可选方案

### 方案 A：沿用根目录 `PySide2`

优点：

- 仓库里已经带了 `PySide2`
- 与现有 Qt5 运行时更贴近

缺点：

- 当前这套 `PySide2` 绑定 `python36.dll`
- 不能直接跑在当前主 Python 或仓库内置 `python313` 上
- 真要沿用，意味着还要恢复一套兼容的 Python 3.6 运行时策略

### 方案 B：给仓库内置 `python313` 补一套兼容 Qt 绑定

优点：

- 启动策略最干净
- 不依赖桌面默认 Python
- 更符合 `renderdoc-workbench` 的独立工具定位

缺点：

- 需要整理新的 Qt 依赖
- 第一阶段要补运行时安装或打包说明

### 方案 C：`Tauri + 前端`

优点：

- UI 表达力更强
- 后续产品感更强

缺点：

- 技术栈跨度太大
- 会把第一期从“整理现有 Python 能力”变成“重建一套前后端应用”
- 与当前仓库已有资产复用度低

## 4. 第一阶段推荐

第一阶段推荐：

**优先绑定仓库内置 Python 运行时，并使用 `PySide6` 作为 GUI 绑定。**

原因很直接：

- 你已经明确不希望依赖桌面默认 Python
- 工具仓库应该有自洽运行时
- 先把 runtime 归属纠正，比先争论 `PySide2/PySide6` 更重要

也就是说，第一阶段优先级应该是：

1. 低迁移阻力
2. 快速验证架构边界
3. 能把会话控制和 RDC 分析打通

这一步完成后，再继续做界面和 workflow 接线。

## 5. 第二阶段再考虑什么

等第一期结构稳定后，再决定是否进一步升级 GUI 组织方式。

触发升级讨论的条件应该是：

- 目录结构已经稳定
- 旧 server 逻辑已经拆开
- 主窗口工作流已经跑通
- UI 需求开始超过当前控件能力

在这些条件达成前，不建议先做技术栈升级。

## 6. 建议的开工策略

### 第一阶段

- 启动强制使用仓库内置 Python
- 单窗口
- 三栏布局
- 不追求复杂视觉
- 先把 workflow 接起来
- GUI 绑定使用 `PySide6`

还需要先处理下面二选一问题：

1. 继续沿用当前内置 Python/Qt 运行时，并固定一个兼容启动方式
2. 升级/替换为与当前主 Python 版本兼容的 Qt 运行时

### 第二阶段

- 评估是否升级到 `PySide6`
- 评估是否需要更强的状态管理
- 评估是否需要拆更细的面板和对话框

## 7. 当前建议结论

如果你现在就要准备开工，我建议这样定：

- 第一阶段运行时：仓库内置 Python
- 第一阶段 GUI 绑定：`PySide6`
- 第一阶段应用形态：单窗口桌面工具
- 第一阶段目标：打通会话控制、RDC 浏览、分析和导出

但在真正实现 GUI 前，还需要先确认：

- 是否继续固定 `python313 + PySide6`
- 还是恢复一套旧 `python36 + PySide2` 兼容运行时

等第一期功能闭环跑通后，再决定要不要做技术栈升级。
